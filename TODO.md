## `turn_done` ≠ конец воспроизведения — окно доставки открывается слишком рано

**Суть.** `DeliveryWindow` снимает флаг `isResponseActive` на событии `turn_done`
(`backend/src/deliveryWindow/deliveryWindow.ts` — `onTurnDone`). Но в SDK
`turn_done` маппится на `response.done` Realtime API, а это **конец генерации
ответа на сервере, а не конец проигрывания звука у пользователя**.

Realtime API генерирует аудио быстрее реального времени: сервер за ~сотни мс
отдаёт весь PCM, бэк релеит его на фронт, фронт буферизует и играет в реальном
времени (24 кГц). Поэтому в логах `[turn] model response done` приходит заметно
**раньше**, чем дослушан звук.

**Следствие.** В момент `turn_done` band переключается `interrupt → full`, и
`tryDeliverInsight` может заинжектить инсайт (`response.create`), пока юзер ещё
дослушивает предыдущий ответ. Новый ответ встаёт в очередь и играет сразу за
текущим — воспринимается как «модель тараторит без паузы / перебивает сама себя».

**Что можно сделать (когда дойдут руки):**

- Гейтить открытие окна не по `turn_done`, а по **реальному концу
  воспроизведения**. Точный сигнал есть только на фронте (AudioContext знает,
  когда доиграл последний чанк) → слать с фронта `playback_done` обратно на бэк и
  открывать окно по нему.
- Серверное `output_audio_buffer.stopped` — про серверный буфер, для нашего
  фронтового playback тоже неточное; не панацея.
- Минимум без изменения поведения: развести в логах генерацию и воспроизведение,
  чтобы `done` не читался как «звук закончился».

Это изменение **поведения**, не только логов — трогать осознанно.
